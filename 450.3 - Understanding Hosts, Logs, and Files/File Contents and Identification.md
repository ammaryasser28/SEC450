## Introduction

في الـ Section ده هنغوص في طبيعة الملفات على مستوى الـ Bytes، وهنتعلم إزاي نحللها عشان نعرف نوعها ومحتواها ونستخرج منها معلومات إضافية. فهم الملفات على المستوى ده مهارة أساسية، لأن فيه مواقف كتير هتحتاجها فيها. مثلًا، لو جالك ملف PCAP فيه جهاز مصاب عمل طلب GET لرابط معين، الجهاز استقبل ملف معين... إيه هو الملف ده؟ وإزاي نعرف لو خطير ولا لا؟

## محتوى الملف: إيه هو الملف فعلاً؟

على أبسط مستوى، **الملف هو مجرد سلسلة من الـ Bytes**. السلسلة دي ممكن تمثل أي حاجة:

- ممكن تكون نص عادي زي PowerShell Script.
- ممكن تكون برنامج مُترجم (Compiled) زي Executable.

كل نوع ملف (PDF, DOCX, TXT) له خصائص فريدة، والبرنامج اللي بيفتحه لازم يفسّر تسلسل الـ Bytes بطريقته الخاصة عشان يعرضها على الشاشة. لكل نوع طريقة مختلفة تمامًا، وله انعكاسات أمنية مختلفة.

### أداة hexdump

لو عايز تشوف الـ Bytes الخام لأي ملف، تقدر تستخدم أداة زي **hexdump**. المخرجات بتتكون من 3 أعمدة:

| العمود | الوصف |
|------|--------|
| العمود الأيسر | الـ Offset من بداية الملف |
| العمود الأوسط | الـ Bytes ممثلة بصيغة Hex |
| العمود الأيمن | نفس الـ Bytes ممثلة بأحرف ASCII (لو ممكن) |

> [!NOTE]
> لو شغّلت hexdump على ملف PDF عشوائي، هتلاقي أول شوية Bytes فيهم الحروف "PDF" — ده تلميح سهل إننا قدام صيغة PDF حتى لو ماكناش عارفين ده من الأساس.

## التعرف على نوع الملف

### السيناريو

جهاز نزّل ملف من `http://mystery.com/x`. إيه نوع الملف؟ هل هو خطير؟

بعد ما تستخرج الملف بأداة زي **Wireshark**، فيه عدة طرق تقدر تستخدمها:

1. تشغّل أمر **file** في Linux لمحاولة التعرف الأوتوماتيكي على النوع.
2. تعمل hexdump وتدور على **Magic Bytes**.
3. تفحص الـ **Strings** بحثًا عن أدلة أو محتوى متداخل.
4. تعمل تحليل أوتوماتيكي في **Sandbox** أو **Virtual Machine** (Dynamic Analysis).

## استخدام أمر file

أمر **file** في Linux هو أول محاولة سريعة للتعرف على أي ملف. الأداة دي عندها مكتبة كبيرة من الصيغ المعروفة، وبتقدر تحدد نوع الملف بناءً على الـ Bytes بتاعته.

### مثال

لو عندك 6 ملفات من غير امتداد، وشغّلت الأمر:

```bash
file *
```

النتيجة ممكن تكون:

| الملف | النوع |
|------|--------|
| A, F | Windows Executable |
| B | Linux Executable |
| C | PNG Image |
| D | SVG Image |
| E | PDF File |

> [!TIP]
> لو الملف مجرد نص عادي، الأمر file غالبًا هيقوله "ASCII text". الاستثناء بيحصل لما النص يكون لغة سكريبتنج معروفة، زي bash script اللي ممكن يظهر كـ "Bourne-Again shell script, ASCII text executable".

## الـ Magic Bytes

### إزاي أمر file بيشتغل؟

عن طريق التعرف على **Magic Bytes**، وهي معروفة كمان بـ **File Signature**. تقريبًا كل صيغة ملف عندها معيار (Standard) بيحدد إزاي الـ Bytes بتاعتها لازم تكون منظمة، ومن أهم ملامح الهيكل ده وجود **Signifier** في أول شوية Bytes من الملف بيأكد نوعه.

### أشهر الـ Magic Bytes

| نوع الملف | الـ Bytes الأولى (Hex) | ASCII |
|------|--------|---------|
| EXE | 4D 5A | MZ |
| ZIP, DOCX, XLSX, PPTX | 50 4B 03 04 | PK.. |
| GZ | 1F 8B | .. |
| PDF | 25 50 44 46 2D | %PDF- |
| RTF | 7B 5C 72 74 66 31 | {\rtf1 |
| PNG | 89 50 4E 47 0D 0A 1A 0A | .PNG... |

> [!IMPORTANT]
> لاحظ حاجة غريبة: ملفات الـ **ZIP** لها نفس الـ Magic Bytes بتاعة مستندات **Microsoft Office** (DOCX, XLSX, PPTX). ده لأن مستندات الـ Office في الحقيقة هي **ملفات ZIP**! تقدر تغيّر الامتداد، تفك الضغط، وتشوف المكونات الداخلية اللي بنيت بيها الملف.

مش كل الـ Magic Bytes ليها تمثيل ASCII مقروء (الحروف القابلة للطباعة اتحطت Bold في الجدول)، وفي الحالة دي هتحتاج تستخدم **hexdump** عشان تتعرف عليها يدويًا.

## الملفات المتداخلة (Nested Files)

### ليه مانعتمدش على أمر file بس؟

أمر file بيشتغل كويس لما الملف يكون منفصل بالفعل، لكن لو بتشوف تدفق خام (Raw Stream) من الـ Bytes جاي من PCAP، أو ملف متضمّن (Embedded) جوه ملف تاني، أمر file **مش هيقدر يتعرف عليه**. هنا لازم تعتمد على معرفتك بالـ Magic Bytes.

### تقنية الملفات المتداخلة

بعض أنواع الملفات (زي Word doc أو PDF) بتسمح بتضمين ملفات إضافية جواها، وده أسلوب شائع في الـ Phishing. الأدوات زي **foremost**, **binwalk**, و **scalpel** مصممة عشان تدور على الـ Magic Bytes في **أي نقطة** جوه الملف مش بس في البداية، وتقدر تستخرج الملفات المتداخلة أوتوماتيكيًا.

### مثال

في ملف .doc خبيث، ممكن نلاقي Magic Bytes لملف ZIP مضغوط متضمّن جواه. حرفين **PK** بيكونوا أسهل حاجة تلاحظها، متبوعين بـ Bytes غير قابلة للطباعة زي `0x03` و `0x04`.

> [!TIP]
> لو مهتم بالتعمق في الموضوع ده لأغراض تحليل الـ Malware، كورس **FOR610: Reverse Engineering Malware** من SANS بيغطيه بالتفصيل.

## الـ Polyglots

### إيه اللي بيخلي ملف صالح (Valid)؟

الإجابة **مش الاسم أو الامتداد**، لكن **100% بتعتمد على الـ Bytes نفسها**. لو الـ Magic Bytes بتشير إن الملف PDF، والمحتوى بيتبع مواصفات الـ PDF، وبيفتح في PDF Reader، فبكل المقاييس ده ملف PDF.

### لكن ممكن يكون حاجة تانية كمان؟

نعم! ده اللي بيتسمى **Polyglot File** — ملف صالح بأكتر من صيغة في نفس الوقت. مثال طريف: صورة اتنشرت على Twitter، وهي فعليًا صورة شغالة، لكن لو فككتها كملف مضغوط، هتلاقي أرشيف **RAR متعدد الأجزاء** فيه نص لشكسبير جواه!

> [!WARNING]
> رغم إن استخدام الـ Polyglots في سياق أمني مش شائع، إلا إنه ممكن يكون طريقة ذكية جدًا لإخفاء ملف خبيث لو المهاجم قرر يستخدمها. المعايير ممكن تتخلط بحيث نفس تسلسل الـ Bytes يتقرأ بشكل صحيح بأكتر من برنامج في نفس الوقت.

## الـ Unicode والـ Encoding

### الفكرة الأساسية

بما إن أغلب العالم مايستخدمش حروف ASCII الإنجليزية بس، لازم يكون فيه Mapping أوسع للحروف. **Unicode** هو النظام اللي فاز في النهاية، وبيعرّف أكتر من **1.1 مليون حرف** كـ **Code Points**، وكل حرف بيتكتب كـ `U+` متبوع برقم (مثال: A = U+41).

### الفرق بين Code Point والـ Encoding

الـ Code Point هو رقم الحرف نفسه، أما **الـ Encoding** فهو الطريقة اللي بيتحول بيها الـ Code Point لـ Bytes فعلية جوه الملف. من أشهر الـ Encodings:

| Encoding | الوصف |
|------|--------|
| ASCII | حرف واحد = Byte واحدة، بيغطي الحروف الإنجليزية بس |
| UTF-8 | طول متغير (Variable Length)، أقل استهلاك للمساحة |
| UTF-16 | طول متغير، بيستخدم غالبًا 2 Byte لكل حرف |
| UTF-32 | طول ثابت (Constant Length)، 4 Bytes لكل حرف دايمًا |

### مثال عملي

جملة "学SEC450" (学 معناها "دراسة" بالكانجي)، ممكن تتكتب بـ 3 طرق مختلفة تمامًا من الـ Bytes:

```
UTF-8  = e5ada6 53 45 43 34 35 30
UTF-16 = 5b66 0053 0045 0043 0034 0035 0030
UTF-32 = 00005b66 00000053 00000045 00000043 00000034 00000035 00000030
```

| Encoding | طول التمثيل | ملاحظة |
|------|--------|---------|
| UTF-8 | متغير | الحروف الإنجليزية بنفس قيم ASCII، لكن الكانجي محتاج أكتر من Byte |
| UTF-16 | ثابت في المثال ده (2 Byte لكل حرف) | كل الحروف اتمثلت بنفس الطول |
| UTF-32 | ثابت دايمًا (4 Bytes) | أقل كفاءة لكن أسهل تفسير |

> [!IMPORTANT]
> لو حاولت تفك تشفير نص مكتوب بـ UTF-8 باستخدام جدول ASCII أو UTF-16 أو UTF-32 غلط، هتحصل على حروف مشوّهة وغير قابلة للقراءة. لازم تعرف الـ Encoding الصحيح عشان تقرأ النص صح.

## إيه هي الـ Strings

بما إن الملفات مجرد تسلسل Bytes من 0 لـ 255 (0x0 لـ 0xFF بالـ Hex)، معايير الترميز زي **ASCII** بتربط بعض الـ Bytes بحروف قابلة للطباعة.

### التعريف

**String** هي سلسلة متصلة من الحروف القابلة للطباعة، وبتنتهي أول ما تقابل Byte غير قابلة للطباعة.

### مثال

لو الملف فيه الـ Bytes: `[00 01 02 03 74 65 73 74 04 05 06]`، الـ String الوحيدة هتكون "test" لأنها المجموعة الوحيدة من الحروف المتتالية القابلة للطباعة.

### التصنيف: Plaintext مقابل Binary

| النوع | الوصف | أمثلة |
|------|--------|---------|
| Plaintext | كل الملف حروف قابلة للطباعة، فالملف كله "String واحدة طويلة" | txt, vbs, ps1, js |
| Binary | فيه Bytes قابلة للطباعة وفيه Bytes مش قابلة | zip, docx, exe, jpg |

### أمر strings في Linux

بيطبع كل التسلسلات المتصلة من الحروف القابلة للطباعة في ملف معين، عن طريق محاولة فك التشفير بمعايير مختلفة. تشغيله على ملف Plaintext مش هيديك حاجة مفيدة (لأن الملف كله String واحدة)، لكن تشغيله على ملف Binary زي Executable ممكن يكشف Metadata أو URLs أو معلومات مهمة تانية.

## مثال عملي على الـ Strings

لو شغّلت `strings -n 10` (يعني أقل طول للـ String هو 10 حروف) على ملف Windows Executable، هتلاقي مخرجات زي:

| الـ String المكتشفة | الدلالة |
|------|--------|
| "This program cannot be run in DOS mode" | جملة موجودة في كل ملفات EXE، دليل واضح إن الملف executable |
| MSVBVM60.DLL | مكتبة مرتبطة بـ Microsoft Visual Basic Virtual Machine، معناها البرنامج اتعمل بـ Visual Basic |
| "REATTACKED" | String غريبة ممكن تكون فريدة لعائلة Malware معينة، مفيدة للتعرف على ملفات مشابهة لاحقًا |

> [!NOTE]
> جزء كبير من الـ Bytes في أي Executable بيكون غير قابل للطباعة، فمفيش Strings تظهر منه. الأجزاء القابلة للطباعة بس (زي "MZ" في البداية) لو أقل من الحد الأدنى المحدد (10 حروف هنا) مش هتظهر في المخرجات.

## الـ Strings بترتيب Bytes وEncodings مختلفة

### تذكير مهم

مش كل الـ Encodings هي ASCII، ومش كل حرف بياخد Byte واحدة بالظبط! لو استخدمت أمر strings العادي بس، ممكن تفوّت Strings مهمة جدًا مكتوبة بترميز تاني.

### خيار -e في أمر strings

| الرمز | المعنى |
|------|--------|
| s | حروف 7-bit مفردة (ASCII، الافتراضي) |
| S | حروف 8-bit مفردة |
| b | 16-bit Big Endian |
| l | 16-bit Little Endian |
| B | 32-bit Big Endian |
| L | 32-bit Little Endian |

> [!TIP]
> **Little Endian** معناها إن الـ Byte الأقل أهمية (Least Significant Byte) بييجي أولًا. لو استخدمت وضع ASCII بس، ومفيش Strings ظهرت، جرّب الـ Encodings التانية زي `-e l` (16-bit little endian) قبل ما تستسلم — لأن النص ممكن يكون UTF-16، ومع ASCII هتفسّره غلط كأنه مجموعة حروف مفردة مفصولة بـ Null Bytes.

## إزاي الـ Strings بتساعدنا

### أهميتها للـ Static Analysis

كتير من الـ **IOCs (Indicators of Compromise)** بتكون Strings عادية قابلة للطباعة. النظر في Strings بتاعة Executable طريقة ممتازة للتعرف على:

- الملفات اللي ممكن الـ Malware يُنزّلها.
- الـ Domains أو IP Addresses اللي هيتواصل معاها.
- الـ Registry Keys المتأثرة.

> [!IMPORTANT]
> تحليل الـ Strings من غير ما تشغّل البرنامج بيتسمى **Static Analysis** وهو آمن تمامًا. البديل هو **Dynamic Analysis**، وده لما تسيب البرنامج يشتغل فعليًا في Sandbox أو Virtual Machine.

### مميزات إضافية

- بتشتغل كويس على **PCAP files, PDFs, Pictures** وأنواع تانية كتير، مش بس على Executables.
- بتساعدك تكتشف Domains, User-Agents, Cookies, URLs بسهولة من داخل الملفات.

### القيود

- **مش هتشتغل على الملفات المضغوطة (Compressed)**، لأن الضغط بيغيّر البنية الداخلية للملف خالص.

### Strings من الذاكرة (Memory)

فيه طريقة أكتر تقدمًا بتستخرج الـ Strings من الذاكرة (Memory Bytes) لبرنامج شغّال فعليًا. الطريقة دي هي **Dynamic Analysis** لأن البرنامج لازم يكون شغّال.

> [!NOTE]
> لو الـ Malware مُشفّر أو "Packed"، ممكن يظهر Strings في الذاكرة ماكانتش تظهر في الـ Static Analysis العادي. لو شفت Strings في Sandbox معينة (زي hybrid-analysis) ومالقتهاش لما شغّلت أمر strings بنفسك، ده غالبًا لأنها اتستخرجت من الذاكرة أثناء التشغيل الفعلي.

## Summary 

- أي ملف هو مجرد **سلسلة من الـ Bytes**؛ الامتداد (Extension) مالوش معنى حقيقي في تحديد النوع.
- بعض الملفات "Plaintext" (كل الحروف قابلة للطباعة) وبعضها "Binary" (فيها Bytes غير قابلة للطباعة).
- أمر **file** في Linux هو أول أداة سريعة للتعرف على نوع الملف.
- **Magic Bytes** (أو File Signature) هي أول شوية Bytes بتحدد نوع الملف، وممكن تشوفها بأداة زي hexdump.
- الملفات المتداخلة (Nested Files) محتاجة أدوات زي **foremost, binwalk, scalpel** عشان تُكتشف.
- **Polyglot Files** هي ملفات صالحة بأكتر من صيغة واحدة في نفس الوقت.
- **Unicode** بيعرّف الحروف كـ Code Points، والـ **Encoding** (ASCII, UTF-8, UTF-16, UTF-32) بيحدد إزاي بتتحول لـ Bytes فعلية.
- **Strings** هي سلاسل متصلة من الحروف القابلة للطباعة، وأداة **strings** في Linux بتستخرجها.
- لازم تجرب Encodings مختلفة مع أمر strings (باستخدام `-e`) عشان ماتفوّتش أدلة مهمة.
- تحليل الـ Strings هو **Static Analysis** آمن، ومفيد جدًا للتعرف على IOCs بدون تشغيل الملف.
